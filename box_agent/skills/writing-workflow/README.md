# writing-workflow

`writing-workflow` 是一个面向公开 SkillHub 用户的通用写作工作流技能，适用于将零散需求、初稿或资料快速转化为结构清晰、表达稳健、可直接使用的高质量文本。它强调目标导向、读者意识、结构化表达与质量自检，适合在报告、方案、文案、邮件、总结、发言稿等多类职场写作任务中使用。

## 定位

`writing-workflow` 将常见写作流程拆解为“理解目标—规划结构—起草内容—优化表达—质量检查”的协作链路，帮助用户在较短时间内获得更成熟、更专业、更贴近场景的文字成果。

它不是单一的润色工具，也不承担宿主产品里的“专家团”身份；当宿主已选择写作专家或专家团时，本技能作为底层工作流和质量门禁使用。

## 适用场景

适合用于以下公开、通用的写作任务：

- **职场汇报**：周报、月报、项目总结、阶段复盘、述职材料
- **商业文档**：方案说明、项目提案、合作建议、产品介绍、运营计划
- **沟通文本**：邮件、通知、会议纪要、客户沟通话术、内部公告
- **内容创作**：公众号文章、短文案、活动文案、品牌介绍、脚本草稿
- **材料优化**：结构重组、语言润色、逻辑增强、语气调整、内容精简
- **多版本改写**：正式版、简洁版、口语版、管理层版、客户版等

不建议用于：

- 包含敏感隐私、密钥、内部机密或未授权资料的写作任务
- 需要法律、医疗、投资等专业资质背书的最终决策文本
- 需要事实核验但未提供可靠来源的高风险内容发布

## 核心能力

### 1. 写作目标澄清

根据用户输入识别写作目的、目标读者、使用场景、语气要求和交付形式，避免只做表层润色。

### 2. 结构化内容规划

自动梳理文章框架、段落层级、重点顺序和信息展开逻辑，使内容更易读、更有说服力。

### 3. 多类型文本起草

支持从简单需求直接生成完整草稿，也支持根据标题、要点、资料片段或已有文本进行扩展成文。

### 4. 风格与语气控制

可根据场景调整为正式、专业、简洁、亲和、有说服力、适合管理层、适合客户沟通等不同表达风格。

### 5. 内容优化与重写

对已有文本进行逻辑优化、表达润色、冗余压缩、重点强化、标题优化、段落重组和语义校正。

### 6. 质量自检

从完整性、逻辑性、读者匹配度、表达清晰度、风险措辞等维度进行检查，减少低级错误和不合适表达。

## 快速开始

### 基础用法

直接描述你要写什么、给谁看、用于什么场景：

```text
请帮我写一封给客户的项目延期说明邮件，语气要诚恳、专业，说明延期原因、补救措施和新的交付时间。
```

### 带资料起草

提供背景信息、关键要点或已有材料：

```text
请基于以下要点起草一份项目周报：
- 本周完成了用户访谈和竞品分析
- 发现核心问题是新手引导不清晰
- 下周计划输出改版方案并评审
- 风格要求：简洁、适合发给管理层
```

### 优化已有文本

粘贴原文并说明优化目标：

```text
请优化下面这段汇报文字，使逻辑更清楚、表达更专业，控制在 300 字以内：
【粘贴原文】
```

### 生成多个版本

```text
请把这段产品介绍分别改写成三个版本：
1. 面向客户的正式版
2. 面向内部团队的简洁版
3. 适合公众号开头的吸引版
```

## 输入输出示例

### 示例一：邮件起草

**输入**

```text
帮我写一封客户沟通邮件，说明项目验收时间需要从本周五调整到下周三。原因是最后一轮兼容性测试还需要补充。语气要专业、负责，不要显得推卸责任。
```

**输出示例**

```text
主题：关于项目验收时间调整的说明

尊敬的客户您好：

为确保本次项目交付质量，我们在最终检查阶段发现仍需补充一轮兼容性测试，以进一步确认不同使用环境下的稳定性表现。基于对交付结果负责的原则，建议将原定于本周五的验收时间调整至下周三。

在此期间，我们将完成补充测试、问题复核及交付材料整理，并及时同步进展。如您方便，我们也可以提前安排一次简短沟通，说明当前测试情况和后续安排。

感谢您的理解与支持，我们会继续按高质量交付标准推进后续工作。
```

### 示例二：汇报优化

**输入**

```text
请优化这段项目汇报：
本周做了很多事情，包括开会、调研、看竞品，也发现了一些问题。下周会继续推进方案，争取早点做出来。
```

**输出示例**

```text
本周项目重点推进了三项工作：一是完成多轮内部沟通，进一步明确业务目标与关键约束；二是开展用户调研，梳理当前流程中的主要痛点；三是完成竞品分析，为后续方案设计提供参考。

目前初步发现，新手引导、信息层级和关键操作路径仍有优化空间。下周将基于调研和竞品结论形成初版改进方案，并组织评审，推动方案尽快进入落地阶段。
```

### 示例三：多版本改写

**输入**

```text
请将“我们的系统可以帮助团队提高协作效率”改写成正式版、营销版和口语版。
```

**输出示例**

```text
正式版：本系统旨在提升团队协同效率，优化跨角色沟通与任务推进流程。

营销版：让团队协作更顺畅，让任务推进更高效，一套系统提升整体工作效率。

口语版：这个系统主要是帮团队少一点沟通成本，把事情推进得更快、更顺。
```

## 质量亮点

- **目标导向**：优先围绕写作目的和读者需求组织内容，而不是机械生成文本。
- **结构清晰**：强调标题、段落、层级和逻辑顺序，适合职场和商业场景使用。
- **风格可控**：可根据不同对象和渠道调整语气，减少“千篇一律”的表达。
- **可直接复用**：输出尽量接近可复制、可发送、可发布的成稿状态。
- **支持迭代**：适合多轮优化，可继续要求压缩、扩写、改语气、换结构或生成多个版本。
- **风险意识**：对过度承诺、模糊责任、敏感表达和不合适措辞保持审慎。

## FAQ

### Q1：这个技能适合从零写作，还是只适合润色？

两者都适合。你可以只提供一个写作目标，让它从零起草；也可以提供已有文本，让它进行优化、重写或改写。

### Q2：如果我不知道该怎么描述需求怎么办？

可以先说明三个基本信息：写什么、给谁看、希望达到什么效果。例如：“写一份项目总结，给管理层看，希望突出成果和下一步计划。”

### Q3：它能保证事实完全准确吗？

不能替代事实核验。对于涉及数据、政策、合同、法律、医疗、投资等内容，建议提供可靠来源，并在发布前进行人工审核。

### Q4：是否可以处理很长的资料？

可以处理较长材料，但建议尽量提供清晰的重点、目标和输出要求。对于特别长或复杂的资料，可分段处理，再统一整合。

### Q5：可以指定固定格式吗？

可以。你可以明确要求输出为邮件、表格、提纲、Markdown、会议纪要、汇报稿、发言稿等格式。

### Q6：如何让输出更贴近我的风格？

可以提供一段参考文本，并说明希望保留的风格特点，例如“简洁、有管理层视角、不夸张、不口语化”。

### Q7：是否适合直接发布到外部渠道？

多数通用文案可作为发布初稿或接近成稿使用，但外部发布前仍建议进行事实、合规、品牌语气和敏感信息检查。

## 发布注意事项

公开发布或在 SkillHub 上架时，建议注意以下事项：

1. **避免包含隐私和密钥**：不要在示例、配置或说明中包含个人身份信息、API Key、访问令牌、内部账号、客户隐私或未授权资料。
2. **保持示例通用化**：示例应使用公开、虚构或泛化场景，避免暴露真实业务细节。
3. **明确适用边界**：对法律、医疗、投资、合规等高风险场景，应提示用户进行专业审核。
4. **注明人工复核建议**：用于正式外发、合同、公告、品牌传播等场景时，建议发布前进行人工确认。
5. **避免过度承诺**：不要宣称技能可以保证事实完全准确、自动完成合规审查或替代专业人员判断。
6. **保留可迭代空间**：鼓励用户通过补充背景、调整语气、指定格式和多轮反馈获得更贴合需求的结果。

## 打包边界

发布包应只包含技能运行和审核所需文件，例如 `SKILL.md`、`AGENTS.md`、`README.md`、`LICENSE`、`CHANGELOG.md`、`skillhub.json`、`package.json`、`examples/`、`references/`、`tests/`、`docs/` 和必要的 `reports/`。不要把本地状态、系统缓存、编辑器临时文件或运行日志打进发布包。

建议按 `.skillignore` 排除本地文件，例如 `.omx/`、`.DS_Store`、`.git/`、缓存目录和日志文件。
