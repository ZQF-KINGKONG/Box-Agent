# 「写作工作流」Skill 工作流参考文档

## 1. 文档定位

本文用于定义「写作工作流」Skill 包在不同写作场景下的标准工作流参考，包括触发条件、执行步骤、质量门禁与交付格式。目标是让 Skill 在面对快速成文、深度原创、素材转化、个人风格复刻、事实核查、多平台改写以及外部证据入口时，能够稳定、可控、可复用地完成写作交付。

## 2. 通用工作原则

### 2.1 北极星目标

在所有工作流中，优先保证：

1. 内容满足用户的真实发布或使用目标；
2. 观点清晰、结构完整、语言可读；
3. 事实、引用、风险表达可追溯、可降级；
4. 输出格式直接可用，减少用户二次整理成本。

### 2.2 通用输入识别

每次进入写作任务时，应先识别以下要素：

- 写作目标：发布、汇报、转述、营销、评论、知识科普、个人表达等；
- 目标读者：内部团队、管理层、客户、公众、专业人群、平台算法推荐受众等；
- 内容类型：文章、短文、长文、脚本、公众号、朋友圈、小红书、视频口播、邮件、报告等；
- 风格要求：理性、犀利、温和、故事化、学术化、管理者视角、个人 IP 风格等；
- 素材来源：用户口述、文档、链接、授权知识库片段、授权阅读摘录、已有文章、数据表等；
- 风险等级：是否涉及事实判断、法律医疗金融、商业敏感、品牌声誉、第三方评价等。

### 2.3 通用交付原则

- 能直接成文时，优先交付完整版本；
- 信息不足但不影响主体写作时，可基于合理假设起草，并标注待确认点；
- 涉及事实风险时，必须给出证据状态或降级表达；
- 涉及平台发布时，必须适配平台语气、长度、标题与开头；
- 不主动编造事实、数据、出处、引语、书籍内容或第三方评价。

## 3. 工作流一：快速成文

### 3.1 触发条件

当用户提出以下需求时，触发快速成文工作流：

- “帮我写一篇/一段/一个稿子”；
- 已给出主题、观点或大致方向，希望快速获得可用初稿；
- 时间优先于深度调研；
- 目标是会议发言、朋友圈、公众号短文、邮件、内部通知、短视频口播稿等；
- 用户没有要求大量事实核查或原创深挖。

### 3.2 执行步骤

1. **识别写作目标**
   - 判断内容用途、读者、篇幅、语气、平台；
   - 若信息缺失，使用默认方案：结构清晰、表达稳健、适合直接修改发布。

2. **提炼核心观点**
   - 将用户输入压缩为 1 个中心观点；
   - 补充 2-4 个支撑论点；
   - 避免主题发散。

3. **生成结构**
   - 标题或主题句；
   - 开头：提出问题、场景或观点；
   - 正文：分层展开；
   - 结尾：总结、行动建议或情绪收束。

4. **起草正文**
   - 按目标平台控制长度；
   - 保持语言自然、准确、有节奏；
   - 对不确定事实使用模糊或条件表达。

5. **快速自检与润色**
   - 检查是否跑题；
   - 检查段落衔接；
   - 删除空话、套话和重复表达；
   - 确保用户可直接复制使用。

### 3.3 质量门禁

交付前需满足：

- 中心观点明确；
- 不少于一个完整开头、主体和结尾；
- 没有明显事实编造；
- 没有与用户目的冲突的语气；
- 文风自然，不像模板拼接；
- 如有不确定信息，已使用保守表达或标注待确认。

### 3.4 交付格式

建议格式：

```markdown
## 标题

正文内容……

---

可选补充：
- 可替换标题 3 个
- 可压缩版/更正式版/更口语版
- 待确认信息
```

## 4. 工作流二：深度原创

### 4.1 触发条件

当用户提出以下需求时，触发深度原创工作流：

- 要求“深度文章”“原创长文”“观点稿”“战略思考”“行业分析”；
- 需要形成有辨识度的观点，而不是简单改写；
- 用户希望文章具备洞察、框架、案例或推理链；
- 目标用于公众号长文、专栏、演讲稿、管理层表达、品牌内容等。

### 4.2 执行步骤

1. **明确议题边界**
   - 定义文章讨论的问题；
   - 明确不讨论什么；
   - 判断文章面向专业读者还是大众读者。

2. **建立原创观点**
   - 提炼一个主论断；
   - 设计反常识、递进式或框架型观点；
   - 避免停留在“重要、复杂、值得关注”等泛泛判断。

3. **搭建论证框架**
   - 背景：为什么现在要讨论；
   - 问题：核心矛盾是什么；
   - 分析：原因、机制、影响；
   - 判断：趋势或结论；
   - 建议：行动路径或方法论。

4. **补充论据与案例**
   - 优先使用用户提供素材；
   - 需要外部事实时，应标注需核查；
   - 案例不能凭空捏造具体细节。

5. **完成长文初稿**
   - 使用清晰小标题；
   - 每一节服务于主论断；
   - 控制抽象概念密度，加入解释与例子。

6. **深度修订**
   - 检查推理链是否完整；
   - 删除无效铺垫；
   - 强化开头吸引力和结尾余韵；
   - 调整语言节奏。

### 4.3 质量门禁

交付前需满足：

- 有明确且可复述的主论断；
- 结构不是资料堆砌，而是围绕问题展开；
- 每个小标题下都有有效推进；
- 事实性内容可区分“已确认”和“待核查”；
- 没有虚构数据、出处或引语；
- 结尾形成判断、启发或行动建议。

### 4.4 交付格式

建议格式：

```markdown
# 标题

> 核心观点：一句话概括本文主论断。

## 一、问题背景

## 二、核心矛盾

## 三、分析框架

## 四、案例/论据

## 五、结论与建议

---

## 事实与证据状态
- 已由用户提供：……
- 需要进一步核查：……
- 已做保守表达：……
```

## 5. 工作流三：素材变文章

### 5.1 触发条件

当用户提供零散素材并希望整理成文时，触发素材变文章工作流：

- 用户提供会议记录、聊天记录、录音转写、笔记、提纲、截图文字、资料摘录；
- 用户要求“整理成文章”“加工一下”“变成公众号/发言稿/报告”；
- 原始素材结构混乱但信息量较大；
- 用户希望保留原意但提升表达。

### 5.2 执行步骤

1. **素材盘点**
   - 提取主题、人物、事件、观点、事实、数据、金句；
   - 区分核心素材与边缘素材；
   - 标记不清楚或冲突的信息。

2. **确定文章类型**
   - 叙事型、观点型、复盘型、说明型、营销型、报告型；
   - 根据类型选择结构。

3. **重组逻辑**
   - 按时间线、问题线、观点线、用户旅程或因果链重排素材；
   - 将口语碎片转化为书面表达；
   - 保留用户原始表达中的关键语气和判断。

4. **补齐缺口**
   - 对必要的过渡、背景、解释进行补写；
   - 不补写无法确认的事实细节；
   - 对缺失信息用“待补充”或保守表达处理。

5. **成文与润色**
   - 形成完整标题、导语、正文、结尾；
   - 优化段落节奏；
   - 保留有价值原句作为金句或引用。

### 5.3 质量门禁

交付前需满足：

- 主要素材均被合理吸收；
- 没有改变用户原意；
- 结构明显优于原始素材；
- 口语化杂音已清理；
- 不确定内容未被写成确定事实；
- 如删减重要素材，应说明删减原因。

### 5.4 交付格式

建议格式：

```markdown
# 成文稿标题

正文内容……

---

## 素材处理说明
- 已重点吸收：……
- 已合并整理：……
- 未纳入/弱化处理：……
- 待确认信息：……
```

## 6. 工作流四：个人风格复刻

### 6.1 触发条件

当用户要求模仿某个个人写作风格时，触发个人风格复刻工作流：

- 用户提供本人历史文章、朋友圈、发言稿、邮件或笔记；
- 用户明确要求“像我平时的风格”“按这个风格写”“复刻我的语气”；
- 目标是保持个人 IP 一致性；
- 用户指定风格样本并授权用于当前任务。

### 6.2 执行步骤

1. **读取风格样本**
   - 分析句长、节奏、用词、常用结构；
   - 提取表达偏好、价值判断、情绪强度；
   - 识别禁用风格：过度夸张、过度营销、过度书面等。

2. **生成风格画像**
   - 总结为 5-8 条可执行规则；
   - 区分“内容观点”和“表达方式”；
   - 不照搬样本文字。

3. **匹配当前写作目标**
   - 判断原风格是否适合当前平台和读者；
   - 如存在冲突，做轻度适配而非机械模仿。

4. **起草正文**
   - 使用相似表达节奏；
   - 保留用户常用的判断方式；
   - 避免逐句仿写或复制独特段落。

5. **风格一致性检查**
   - 对照风格画像检查；
   - 删除不像用户会说的话；
   - 调整过强或过弱的语气。

### 6.3 质量门禁

交付前需满足：

- 风格相似体现在节奏、结构和表达倾向，而不是复制原文；
- 没有冒用第三方未授权风格进行欺骗性仿写；
- 内容仍符合当前任务目标；
- 语气自然，不生硬堆砌口头禅；
- 如样本不足，应说明只能做轻度风格靠近。

### 6.4 交付格式

建议格式：

```markdown
## 风格画像
- 规则 1：……
- 规则 2：……
- 规则 3：……

## 正文稿

正文内容……

---

## 风格适配说明
- 已保留的风格特征：……
- 为适配平台做出的调整：……
- 样本不足/待确认：……
```

## 7. 工作流五：事实核查与风险降级

### 7.1 触发条件

当写作任务存在事实或合规风险时，触发事实核查与风险降级工作流：

- 涉及具体数据、排名、年份、政策、法律、医学、金融、投资、商业判断；
- 涉及第三方品牌、人物、机构评价；
- 涉及争议事件、公共舆论、负面指控；
- 用户要求“核查一下”“别出错”“降低风险”；
- 原文包含绝对化、攻击性、无法证实的表达。

### 7.2 执行步骤

1. **识别高风险陈述**
   - 数据类：金额、比例、排名、增长率；
   - 事实类：时间、地点、人物、事件；
   - 评价类：最、第一、唯一、必然、骗局、违法等；
   - 建议类：法律、医疗、金融、投资决策。

2. **证据分级**
   - A 级：用户提供的原始证据或权威来源可验证；
   - B 级：多方公开资料一致，但仍需引用；
   - C 级：仅有单一来源或间接说法；
   - D 级：无法验证、疑似推测或主观判断。

3. **核查与标注**
   - 对关键事实进行来源核验；
   - 标注已确认、待确认、无法确认；
   - 对无法核查的内容不写成确定事实。

4. **风险降级改写**
   - 绝对表达改为相对表达；
   - 指控表达改为现象描述；
   - 结论表达改为观察、可能、倾向；
   - 专业建议改为一般性信息提示。

5. **输出风险说明**
   - 列出主要风险点；
   - 说明已如何降级；
   - 提醒仍需用户确认或法务审核的内容。

### 7.3 质量门禁

交付前需满足：

- 高风险事实已被识别；
- 不确定内容没有被强化；
- 绝对化表述已检查；
- 涉及法律、医疗、金融等场景时，避免提供确定性专业结论；
- 如无法联网或无法获得证据，应明确说明证据不足；
- 保留文章表达效果，不因降级变得含糊无意义。

### 7.4 交付格式

建议格式：

```markdown
## 改写后安全版本

正文内容……

---

## 风险核查表
| 原表述 | 风险类型 | 证据状态 | 建议处理 | 已改写为 |
|---|---|---|---|---|
| …… | 数据/评价/法律等 | 已确认/待确认/无法确认 | 保留/降级/删除 | …… |

## 仍需确认
- ……
```

## 8. 工作流六：多平台发布改写

### 8.1 触发条件

当用户希望将同一内容发布到不同平台时，触发多平台发布改写工作流：

- 用户要求“改成公众号/小红书/朋友圈/视频号/微博/知乎/抖音口播”；
- 用户已有长文，需要拆分或适配；
- 用户希望一稿多发；
- 用户要求生成标题、摘要、封面文案、口播稿或短帖。

### 8.2 执行步骤

1. **识别源内容核心**
   - 提取主观点、关键故事、核心论据、行动号召；
   - 判断哪些内容可跨平台复用，哪些需要删改。

2. **匹配平台规则**
   - 公众号：结构完整、标题强、开头抓人；
   - 朋友圈：短、真诚、有个人态度；
   - 小红书：场景感、经验感、标题关键词清晰；
   - 视频号/抖音：口语化、节奏快、前三秒有钩子；
   - 知乎：问题意识、论证充分、可信度强；
   - 微博：观点鲜明、信息密度高、适合转发。

3. **平台化改写**
   - 调整标题；
   - 重写开头；
   - 调整段落长度；
   - 增删案例和行动号召；
   - 避免同一文本机械复制到所有平台。

4. **生成发布组件**
   - 标题备选；
   - 摘要/导语；
   - 正文；
   - 标签或话题；
   - 封面文案或口播开场。

5. **平台一致性检查**
   - 核心观点是否一致；
   - 平台语气是否适配；
   - 是否存在平台敏感或误导性表达。

### 8.3 质量门禁

交付前需满足：

- 每个平台版本不是简单截断，而是重新适配；
- 标题、开头与正文风格一致；
- 核心观点未被改偏；
- 平台长度、语气、互动方式合理；
- 风险表达在各平台版本中保持一致降级。

### 8.4 交付格式

建议格式：

```markdown
# 多平台发布版本

## 公众号版
- 标题：……
- 摘要：……
- 正文：……

## 朋友圈版
正文……

## 小红书版
- 标题：……
- 正文：……
- 标签：……

## 视频口播版
- 前 3 秒钩子：……
- 口播正文：……
- 结尾引导：……

## 发布注意事项
- ……
```

## 9. 工作流七：授权知识库素材入口

### 9.1 触发条件

当用户提供或引用授权知识库、项目材料或笔记系统内容作为写作依据时，触发授权知识库素材入口工作流：

- 用户说明“参考我的知识库素材”“从这些笔记里找资料”“基于授权材料写”；
- 输入包含用户提供或授权的文本、摘要、链接、知识库片段、项目材料或检索结果；
- 用户希望将授权素材整理成文章、报告、观点或发布内容；
- 用户要求保留素材依据。

### 9.2 执行步骤

1. **接收与归类素材**
   - 区分原文、摘要、评论、标签、检索结果；
   - 标注素材来源名称、时间、作者或上下文；
   - 判断素材是否足以支撑写作目标。

2. **提取可用信息**
   - 提炼事实、观点、案例、数据、引用；
   - 区分素材中的原始信息与二次解读；
   - 标记证据强弱。

3. **构建文章框架**
   - 以用户写作目标为主，不以素材顺序为主；
   - 将授权素材放入对应论点下；
   - 避免资料堆叠。

4. **证据化写作**
   - 对关键事实保留来源标记；
   - 对素材未覆盖的推论使用保守表达；
   - 不把二次摘要误写成原始事实。

5. **输出素材使用说明**
   - 说明用了哪些素材；
   - 哪些素材未使用；
   - 哪些结论仍需补证。

### 9.3 质量门禁

交付前需满足：

- 授权素材与正文观点之间有明确对应关系；
- 未将未经确认的二次摘要当作权威原文；
- 引用、数据、事实表达可追溯；
- 未因素材丰富而牺牲文章主线；
- 对缺证结论已做降级。

### 9.4 交付格式

建议格式：

```markdown
# 正文稿

正文内容……

---

## 授权素材使用说明
| 素材名称/编号 | 用途 | 支撑的观点 | 证据强度 | 处理方式 |
|---|---|---|---|---|
| …… | 背景/案例/数据/引用 | …… | 强/中/弱 | 引用/改写/未采用 |

## 待补充证据
- ……
```

## 10. 工作流八：授权阅读材料证据入口

### 10.1 触发条件

当用户希望使用授权阅读材料中的书籍、摘录、笔记或书摘作为写作证据时，触发授权阅读材料证据入口工作流：

- 用户要求“参考这些读书摘录”“用这本书的观点”“基于我的笔记写”；
- 用户提供阅读摘录、书名、章节、笔记、评论或授权材料片段；
- 用户希望文章带有书籍证据、阅读感、知识背书；
- 用户要求核验书籍是否可读或是否适合引用。

### 10.2 执行步骤

1. **确认证据形态**
   - 书名、作者、版本、章节；
   - 用户划线、笔记、摘录；
   - 用户自己的理解或评论；
   - 是否需要核验书籍版本、出处或可访问状态。

2. **提取书籍观点**
   - 将书摘转化为可写作的观点；
   - 区分作者原意、用户理解和写作延伸；
   - 不扩大书中观点适用范围。

3. **确定引用方式**
   - 直接引用：适合短句、金句，需保持准确；
   - 间接转述：适合概念、框架、案例；
   - 阅读感表达：适合个人文章或朋友圈；
   - 证据背书：适合观点文或报告。

4. **融入正文**
   - 引用服务于文章观点，而不是堆书摘；
   - 给出必要解释；
   - 避免“读书笔记式拼贴”。

5. **证据与版权风险处理**
   - 控制直接引用长度；
   - 不输出大段受版权保护文本；
   - 不伪造页码、章节或原文；
   - 若只知道书名但无摘录，应避免编造书中内容。

### 10.3 质量门禁

交付前需满足：

- 书籍观点、用户观点、文章观点三者边界清楚；
- 直接引用准确且不过量；
- 未虚构书中原文、章节、页码或作者观点；
- 引用内容与文章论点强相关；
- 如书籍版本、出处或可访问状态未核验，应明确说明。

### 10.4 交付格式

建议格式：

```markdown
# 正文稿

正文内容……

---

## 授权阅读材料证据使用说明
| 书名 | 作者 | 证据类型 | 使用位置 | 处理方式 | 备注 |
|---|---|---|---|---|---|
| …… | …… | 划线/笔记/书摘/书名 | 第 X 节 | 直接引用/转述/启发 | …… |

## 证据边界
- 作者原意：……
- 用户理解：……
- 本文延伸：……
- 待核验：……
```

## 11. 组合工作流建议

实际任务中，多个工作流可能同时触发。建议按以下优先级组合：

1. **有事实风险时**：先执行“事实核查与风险降级”，再进入成文或改写；
2. **有大量素材时**：先执行“素材变文章”或“授权知识库素材入口”，再决定是否深度原创；
3. **有个人风格要求时**：先抽取风格画像，再进行成文；
4. **要多平台发布时**：先生成主稿，再做平台化改写；
5. **有授权阅读证据时**：先确认证据边界，再融入文章。

推荐组合示例：

| 用户需求 | 推荐工作流组合 |
|---|---|
| 根据一堆会议纪要写公众号 | 素材变文章 → 快速成文/深度原创 → 事实降级 |
| 按我的风格写一篇行业观点文 | 个人风格复刻 → 深度原创 → 事实核查 |
| 把长文改成小红书和视频口播 | 多平台发布改写 → 风险降级 |
| 基于授权知识库资料写分析文章 | 授权知识库素材入口 → 深度原创 → 事实核查 |
| 用授权阅读摘录写朋友圈 | 授权阅读材料证据入口 → 快速成文 → 风格适配 |

## 12. 总体质量门禁清单

所有工作流最终交付前，应进行以下检查：

- **目标一致性**：是否服务于用户最初写作目标；
- **结构完整性**：是否有清晰开头、主体、结尾或对应格式；
- **观点清晰度**：读者是否能一句话复述核心观点；
- **事实可靠性**：事实、数据、引用是否有来源或已降级；
- **风格适配度**：是否符合用户、平台和读者语境；
- **风险控制**：是否避免绝对化、攻击性、虚假背书和专业误导；
- **可交付性**：是否能直接复制、发布、提交或继续编辑。

## 13. 标准交付格式总览

根据任务复杂度，交付可采用以下三类格式。

### 13.1 轻量交付

适用于快速成文、短帖、邮件、朋友圈：

```markdown
## 成稿

正文内容……

## 可选标题
1. ……
2. ……
3. ……
```

### 13.2 标准交付

适用于公众号、长文、素材整理、平台改写：

```markdown
# 标题

正文内容……

---

## 交付说明
- 适用场景：……
- 风格说明：……
- 待确认信息：……
```

### 13.3 风险型交付

适用于事实敏感、品牌敏感、法律医疗金融相关内容：

```markdown
# 安全改写版本

正文内容……

---

## 风险与证据说明
| 内容点 | 风险 | 证据状态 | 处理方式 |
|---|---|---|---|
| …… | …… | …… | …… |

## 建议用户确认
- ……
```

## 14. 不应执行的行为

「写作工作流」Skill 在任何工作流中均不应：

- 编造不存在的事实、数据、出处、案例、引语；
- 将无法确认的内容写成确定事实；
- 大段复制受版权保护文本；
- 未经授权复刻第三方在世作者或公众人物的独特风格用于欺骗性发布；
- 为法律、医疗、金融等高风险事项提供确定性专业结论；
- 为追求标题党而扭曲正文含义；
- 在多平台改写中改变原文核心立场。

---

## 15. 最小可用执行路径

当任务信息很少但用户希望尽快得到结果时，可采用最小可用路径：

1. 识别用户要写什么；
2. 默认选择快速成文；
3. 若发现事实风险，自动做风险降级；
4. 输出一版可直接使用的成稿；
5. 附上 2-3 个可选标题或待确认点。

最小交付格式：

```markdown
## 成稿

正文内容……

## 可选标题
1. ……
2. ……
3. ……

## 待确认
- ……
```
